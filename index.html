LangGraph

1.LangGraph简介

  • 定义：LangGraph是一个用于开发由大语言模型LLM提供支持的应用程序的框架[用LLM快速构建本地应用的框架]

  • LangGraph构建应用的标准流程：
    • 第一步—Prebuilt agents：构建Agent智能体【LangGraph应用基础】
    • 第二步—LangGraph framework：构建LangGraph应用【LangGraph核心】以Graph图的方式将多个Agent智能体整合一体
    • 第三步—LangGraph Platform：通过LangGraph Platform平台部署应用【商业化平台】标准化的形式部署LangGraph应用，并提供测试、监控、分析等功能
  • 安装依赖：

==第一步—-Prebuilt agents构建Agent智能体==

0.Agent智能体介绍

  • 定义：将大语言模型的各种功能，封装成独立的整体。

  • 好处：Agent构建完成之后，未来我们有什么问题都直接给Agent处理，不需要过多关注Agent的细节。

  • 实现：Agent需要两大能力—①拥有封装与大模型交互的所有基础能力(访问大模型、调用Tools、保存ChatMemory)等基础能力，可以独立完成一系列基于大模型构建的任务②我们可以随时干预Agent执行进度，对关键步骤随时做出调整。

2.工具调用-tools属性

Tools⼯具机制是⼤语⾔模型中的—个重要机制，它可以让⼤模型调⽤外部⼯具，从⽽实现更加复杂的功能。

通常，—个完整的⼯具调⽤流程需要以下⼏个步骤：

  • 1.客户端定义⼯具类，实现⼯具的功能【定义工具类】
  • 2.客户端请求⼤语⾔模型，带上问题以及⼯具的描述信息【大模型添加问题+工具描述】
  • 3.⼤语⾔模型综合判断问题，并决定是否调⽤⼯具【大模型去判断是否使用】
  • 4.如果⼤语⾔模型判断要使⽤⼯具，就会向客户端返回—个带有tool_calls⼯具调⽤信息的AiMessage【大模型告诉客户端我要哪个】
  • 5.客户端根据⼯具调⽤信息，调⽤⼯具，并将结果返回给⼤语⾔模型【返回工具结果】
  • 6.⼤语⾔模型根据⼯具调⽤的结果，⽣成最终的回答。【大模型整合工具结果和问题综合给出答案】

3.会话记忆

LangGraph实现消息记录的流程完整的封装到了Agent中

LangGraph将消息记忆分为短期记忆与长期记忆

  • 短期记忆[CheckPoint]：Agent内部记忆，用于当前对话中的历史记忆信息。【会话级别】
  • 长期记忆[Store]：Agent外部记忆，用于第三方存储长久的聊天信息。【用户级别/应用级别】
定位 级别 关注点
短期记忆 Agent内部记忆 会话级别[小内存] 需要频繁清理内存，对已有消息记录总结，减少内存占用
长期记忆 Agent外部记忆 用户级别/应用级别[大内存] 需要关注对已有消息进行检索

1.大模型消息持久化

【也是类似第一步中3.会话记忆拥有短期记忆和长期记忆，都可以支持内存/数据库持久化保存】

2.人类监督Human-in-the-Loop

LangGraph中也可以通过中断任务(interrupt)，等待确认(Command)的方式来实现过程干预，能更好减少大语言模型的结果不稳定给任务带来的影响。

实现人类干预时，需要注意：

  • ①必须指定一个checkpoint短期记忆，否则无法保存任务状态
  • ②执行Graph任务，必须指定一个带有thread_id的配置项(指定线程ID),这样才能通过线程ID指定恢复线程【搭配短期记忆】
  • ③执行Graph任务，通过interrupt()方法中断任务，等待确认—【时间过长就无法恢复任务】
  • ④在人类确认之后，使用Graph提交一个resume=True的Command指令，恢复任务并继续执行— 【中断和恢复需要相同的thread_id，通常应用中会单独生成一个随机的thread_id】

3.时间回溯Time Travel[从某个Node重演]

由于LLM回答问题的不确定性，因此基于LLM构建的应用也是充满不确定性 —> 有必要进行更精确的检查[当某一个步骤出现问题时，才能及时发现问题，并从发现问题的那个步骤进行重演] —> LangGraph提供了Time Travel时间回溯功能，可以保存Graph的运行过程，并可以手动指定从Graph的某一个Node开始重演

具体实现时，需要注意：

  • ①运行Graph时，需要提供初始的输入消息
  • ②运行Graph时，指定thread_id线程ID + checkpoint短期记忆【执行后在每一个Node执行后，生成一个check_point_id】
  • ③指定thread_id和check_point_id进行任务重演【重演前，可以更新state/不指定更新state】-更新状态update_state()方法

MCP

1.MCP介绍

  • 定义：MCP（Model Context Protocol）模型上下文协议。专门用于与AI大语言模型进行交互的协议。本质上，MCP协议只是允许应用程序以一种统一的方式(类似统一的USB接口)向大语言模型提供Function Call函数调用。

1.1 工作机制

步骤①人类通过用户框输入问题给Agent，问题就被称之为User Prompt

步骤②Agent通过MCP协议访问MCP Server获取所有Tool信息，并且返回给Agent

步骤③Agent整合[第一步的User Prompt+第二步的tool信息(要么普通发送/要么按照更严格的Function Calling格式)]发送给AI模型

步骤④AI模型按照更严格的Function Calling告诉Agent我想请求web_browse工具

步骤⑤Agent根据MCP协议调用web_browse工具然后把结果返回给Agent

步骤⑥Agent将结果给AI模型

步骤⑦AI模型通过学习和整理返回出结果”多喝热水”

步骤⑧Agent将结果返回给用户

1.2 MCP和工具对比

  • 工具到MCP：中心化，封闭生态 –> 去中心化，开发协议

工具：更适合公共服务，需要集中计算资源，控制服务质量和商业化等

MCP：更适合公司内部数据库、敏感数据处理等

4.Tools工具机制(扩展模型能力边界)

AI⼤模型的强⼤能⼒来源于他学过的知识，但是，模型学习到的知识是有限的，如何扩展模型的能⼒边界呢？

  • 答：Tools⼯具机制就是解决这个问题的重要机制。它是让AI⼤模型去调⽤外部的API接⼝，去获取外部数据，然后让AI⼤模型去使⽤这些数据，从⽽扩展模型的能⼒边界。Tools⼯具机制是现在⼏乎所有⼤模型都⽀持的⼀种机制，也是基于⼤模型构建本地应⽤的关键。
  • 原来的大模型(引入的原因)：

很明显，大模型是无法获取实时时间，因为没有现成资料能实时告诉大模型当前的日期，所以给出的答案大概率也是一个不靠谱的答案。理解”今天”对于大模型是一个比较困难的事情。但是，对于我们人类来说却是很容易理解的问题。那么，我们可以给大模型加一些外援，让大模型获取”今天”相关信息，并且返回结果。—这个外援就是Tools工具机制。

根据LangChain我们可以看到，Tools工具机制几乎所有主流大模型都支持，也是基于大模型构建本地应用的关键。

———————-RAG(检索增强生成)———————-

RAG智能客服系统：将企业内部的多个手册整理成一个知识库，允许用户以自然语言方式向客服系统询问知识库相关问题。 智能客服系统可以理解用户题,并基于内部知识库以及LLM大模型理解能力给出用户想要的答案。

适用范围：更适合处理客户自己的，不太适合对互联网公开的资料!!

0.引入原因

AI大模型的强大能力来源于他学习过的知识，但是，模型学习到的知识是很有限的，如何学习”具体领域的业务知识”？

  • 答：①微调 ②RAG检索增强生成
  • 原来的大模型(引入的原因)：

很明显，大模型能给出答案，但是没有学习专门某个APP的业务知识，自然是无法针对APP的常见问题说明，给出比较理想的答案。

那么，如何让大模型能够”学习”APP的业务知识？典型方法有两种：

  • ①微调：让大模型回炉进行一次学习。【在deepseek的预训练模型基础上，投入少量的资源，让大模型继续学习，从而达到更好的理解能力】

  • ②RAG：询问大模型问题时，先检索出和问题相关的参数信息，将[问题+参考信息]一起输入给大模型，让大模型给出更好的答案。【我们向一个专家询问相关业务问题时，带上业务手册。这样专家就不需要提前学习额外知识，只需要结合业务手册就可以快速给出针对性帮助】

微调[大模型补脑子] RAG[大模型锻炼四肢]
原理 大模型回炉重造 外挂知识库，实时检索相关信息，(问题+相关内容)一起给大模型
优点 能学习到新知识，理解能力和针对性更强 不需要提前学习额外知识，成本低
缺点 成本高、难度大[容易倒退] 不是去学习，而是用到了补充
知识更新 困难[重新训练] 容易[只更新知识库]
实时性 延迟[训练需要时间] 实时[立即生效]
可解释性 差[黑盒] 强[可查看检索来源]
使用场景 专业术语、固定流程 政策文档(会快速更新)、产品知识库

通常，RAG和模型微调可以混合使⽤的。RAG是给⼤模型锻炼四肢，⽽模型微调是给⼤模型补脑⼦，两者相辅相成。

3.Quartz基本实现原理

Quartz 任务调度的核心元素是 scheduler, trigger 和 job； trigger 和 job 是任务调度的元数据，scheduler 是实际执行调度的控制器。

  • Scheduler：**任务调度器【组合触发器和任务，就是组装任务该怎么执行】
  • Trigger：触发器【定义任务调度的时间规则】
  • Job：被调度的任务

【注意：一个Job可以被多个Trigger关联，一个Trigger只能关联一个Job；一个任务调度器可以绑定多个任务】

3.1 Scheduler任务调度器[三种]

Scheduler：对任务进行调度和管理

在Quartz中，Scheduler由SchedulerFactory工厂创建，主要提供以下两种工厂实现：

  • DirectSchedulerFactory（使用较麻烦，需要进行编码配置）

  • StdSchedulerFactory（✔）

Scheduler主要有三种：

  • RemoteMBeanScheduler

  • RemoteScheduler

  • StdScheduler（✔）

举例：

3.2 Trigger触发器[四种]

Trigger：调度时间的元素，定义按照什么时间规则去执行任务

在Quartz中，主要提供四种类型的Trigger:

  • SimpleTrigger：使用简单便捷【简单任务】

  • CronTirgger：使用cron表达式定义触发任务的时间规则【复杂任务，整合3和4】

  • DateIntervalTrigger：日期周期的规则的触发器

  • NthIncludedDayTrigger：排除指定时间周期和日期的触发器

3.2.1 SimpleTrigger[基于精准时间]

SimpleTrigger可以满足的调度需求是：

①在具体的时间点执行一次

  • 延时任务：用户提交订单之后启动任务，延时30min检查订单状态，如果未支付则取消订单释放库存
  • 定时任务：在指定的时间点调度执行任务

②在具体的时间点执行 + 执行若干次[以指定的间隔重复]

  • 循环任务：每个指定的时间周期执行一次任务，任务可以（无限循环、循环指定次数、循环执行到指定的时间点）

举例：

其中一些参数设置：

  • 重复次数：0/正整数/常量SimpleTrigger.REPEAT_INDEFINITELY

  • 重复的间隔(ms)：0/正数[long型] —设置为0，trigger会以重复次数并发执行(或者以scheduler可以处理的近似并发数)

  • endTime属性的值会覆盖设置重复次数的属性值 —创建一个trigger，在终止时间之前每隔10秒执行一次，你不需要去计算[开始时间-终止时间]间的重复次数，只需设置终止时间并将重复次数设置为REPEAT_INDEFINITELY。

3.2.2 CronTrigger[基于时间表]

CronTrigger：基于日历的概念，而不是按照SimpleTrigger的精确指定间隔

使用CronTrigger，您可以指时间表，例如“每周五中午”或“每个工作日和上午9:30”，甚至“每周一至周五上午9:00至10点之间每5分钟”和1月份的星期五“。

举例：5秒后开始，每3秒执行一次，截止到今天15:11分

3.2.3 Misfire策略[触发器异常]

前提：调度器Scheduler绑定触发器Trigger和任务Job，在执行过程中
异常情况：①调度器出现故障停止了 ②线程池中没有充足的线程执行现在的任务
结果：导致触发器Trigger触发周期到了但是任务没有执行的情况【这就叫Misfire，其实就是异常了–类似于线程池的拒绝策略】

一些配置：【==代码中，在.withSchedule(内部配置)==】

支持持久化的Trigger：通过指定Misfire策略，来定义其恢复之后的执行策略

所有类型的Trigger：Quartz指定了默认的Misfire策略—MISFIRE_INSTRUCTION_SMART_POLICY【立即执行misfire的任务，然后按照计划继续往后执行】

3.5 Quartz存储与持久化[JobStore]

==JobStore存储什么，调度器就执行什么==

JobStore:存储调度器的”工作数据”：任务(job)，触发器(trigger)，数据(JobDataMap就是一些序列化数据)

3.5.1 两种作业类型

Quartz提供了两种作业类型：

  • RAMJobStore:基于内存存储调度器的工作数据

    • 优点：速度快
    • 缺点：不支持持久化

  • JDBCJobStore:基于数据库存储调度器的工作数据

    • 优点：支持持久化；支持Quartz集群
    • 缺点：速度慢
RAMJobStore【默认】 JDBCJobStore
基于模式 内存 数据库
速度 快 慢
持久化 × √
集群 × √[配置相同数据源，保证数据读写一致]
系统崩溃重启执行计数器 从0开始 从之前次数继续执行

案例：我们要执行100次任务，当执行了40次时系统崩溃了，系统重启时任务的执行计数器默认会从0开始。这种策略能够满足多数业务场景需求；但是在某个特定的场景中我们需要继续之前的任务执行，我们可以通过对JobStore进行持久化来实现。
