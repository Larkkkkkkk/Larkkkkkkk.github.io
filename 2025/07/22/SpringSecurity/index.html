SpringSecurity

  • Spring

1.简介

  • 概要：Spring Security基于Spring框架，提供一套Web应用安全性的完整解决方案。

  • 主页：Spring Security

  • 功能：

    • ①身份认证：验证谁正在访问系统资源，判断用户是否为合法用户
    • ②授权：【用户进行身份认证后】系统会控制谁能访问哪些资源
    • ③防御常见攻击：
      • CSRF
      • HTTP Headers
      • HTTP Requests

  • 特点：

    • ①和Spring无缝整合
    • ②全面的权限控制
    • ③专门为Web开发而设计【旧版本不能脱离Web环境使用，新版本对整个框架进行分层抽取，分为核心模块和Web模块，单独引入核心模块就可以脱离Web环境】
    • ④重量级

6.密码加密算法

6.1 加密方式

  • 明文密码：最初，密码以明文形式存储在数据库中，但是恶意用户可能会通过SQL注入等手段获取到明文密码，会存在数据库数据泄露的情况
  • Hash算法：SpringSecurity的PasswordEncoder接口用于对密码进行单向转换，从而将密码安全存储。这其中涉及Hash算法，例如MD5、SHA-256、SHA-512等。—-【只能加密，不能解密】—如果要比对就只能将用户输入的密码加密之后来判断，防止反向解密[除非暴力破解的方式猜测密码]
  • 彩虹表：恶意用户创建的查找表[一个庞大的、针对各种可能的字母组合预先生成的哈希值集合，有了它可以快速破解各类密码。越是复杂的密码，需要的彩虹表就越大，主流的彩虹表都是 100 GB 以上。目前主要的算法有 LM、NTLM、MD5、SHA1、MYSQLSHA1、HALFLMCHALL、NTLMCHALL、ORACLE-SYSTEM、MD5-HALF。 —-【类似于一个猜测表，如果你只是单纯加密有可能被猜出来】
  • 加盐密码：为了减轻被彩虹表猜中的可能，考虑使用（盐+用户密码）一起经过哈希函数运算，生成一个唯一的哈希。盐以明文形式和用户密码一起存储，当用户尝试身份验证时，盐与用户密码再次经过哈希函数运算，得到的结果与存储的哈希进行比较。由于每个密码对应唯一的盐，彩虹表就不再有效。 —-【我在密码前后都可以加盐，这样结果变化多不让你猜出来】
  • 自适应单向函数：随着硬件的不断发展，单纯的加盐哈希已无法满足安全需求。原因是现代计算机每秒可执行数十亿次哈希计算，使得暴力破解变得轻而易举。 —-【设置“工作因子”故意消耗大量CPU和内存资源，拖慢单次验证速度】，例如bcrypt、PBKDF2、scrypt 和 Argon2。

6.2 SpringSecurity加密方案–自适应单向函数

  • bcrypt算法【默认算法】

BCryptPasswordEncoder使用bcrypt算法对密码进行加密，为了提高密码的安全性, bcrypt算法故意降低运行速度,以增强密码破解的难度。同时BCryptPasswordEncoder自己带有盐值，开发者不需要额外维护一个“盐”字段，使用BCryptPasswordEncoder加密后的字符串就已经“带盐”了，即使相同的明文每次生成的加密字符串都不相同。BCryptPasswordEncoder的默认强度为10，开发者可以根据自己的服务器性能进行调整，以确保密码验证时间约为1秒钟(官方建议密码验证时间为1秒钟，这样既可以提高系统安全性，又不会过多影响系统运行性能)。

  • PBKDF2算法

Pbkdf2PasswordEncoder使用PBKDF2算法对密码进行加密，和前面几种类似，PBKDF2算法也是一种故意降低运算速度的算法，当需要FIPS (全称Federal Information Processing Standard,美国联邦信息处理标准)认证时，PBKDF2 算法是一个很好的选择。

  • argon2算法

SCryptPasswordEncoder使用scrypt算法对密码进行加密，和前面的几种类似，scrypt也是种故意降低运算速度的算法，而且需要大量内存。

  • scrypt算法

SCryptPasswordEncoder使用scrypt算法对密码进行加密，和前面的几种类似，scrypt也是种故意降低运算速度的算法，而且需要大量内存。

————————-授权————————-

1.简介

授权管理的实现在 Spring Security 中非常灵活，可以帮助应用程序实现以下两种常见的授权需求：

  1. 用户-权限-资源：例如张三的权限是“添加用户、查看用户列表”，李四的权限是“查看用户列表”。
  2. 用户-角色-权限-资源：例如张三的角色是“管理员”，李四的角色是“普通用户”；管理员角色可以执行所有操作，普通用户角色只能查看信息。

2.2 用户-角色-权限-资源[RBAC]

RBAC（Role-Based Access Control，基于角色的访问控制）是一种常用的数据库设计方案，它将用户的权限分配和管理与角色相关联。以下是一个基本的 RBAC 数据库设计方案示例：

1.用户表（user）存储用户的基本信息。

列名 数据类型 描述
user_id int 用户 ID
username varchar 用户名
password varchar 密码
email varchar 电子邮件地址

2.角色表（role）存储所有可能的角色及其描述。

列名 数据类型 描述
role_id int 角色 ID
role_name varchar 角色名称
description varchar 角色描述

3.权限表（permission）定义系统中所有可能的权限。

列名 数据类型 描述
permission_id int 权限 ID
permission_name varchar 权限名称
description varchar 权限描述

4.用户角色关联表（user_role）将用户与角色关联起来。

列名 数据类型 描述
user_role_id int 用户角色关联 ID
user_id int 用户 ID
role_id int 角色 ID

5.角色权限关联表（role_permission）将角色与权限关联起来。

列名 数据类型 描述
role_permission_id int 角色权限关联 ID
role_id int 角色 ID
permission_id int 权限 ID
